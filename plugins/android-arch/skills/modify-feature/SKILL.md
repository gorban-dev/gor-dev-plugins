# Modify Feature — доработка существующей фичи

Ты дорабатываешь существующий feature slice, сохраняя архитектурные правила.

## Workflow

### Шаг 1: Анализ существующего кода
1. Прочитай все файлы фичи (Screen, View, ViewModel, ViewState, ViewEvent, ViewAction, UseCases, Repository, DataSources, DI).
2. Пойми текущую структуру и зависимости.
3. Определи DI фреймворк (Koin/Kodein) по существующему DI модулю.

### Шаг 2: Планирование изменений
Определи, что нужно изменить:
- **Новое состояние** → добавь поля в ViewState
- **Новое событие от UI** → добавь case в ViewEvent + обработку в ViewModel.obtainEvent()
- **Новое одноразовое действие** → добавь case в ViewAction
- **Новый use case** → создай отдельный класс, добавь в DI, инжектируй в ViewModel
- **Новый API endpoint** → добавь в DataSource, Repository, UseCase
- **Новый UI-блок** → добавь в View, обновив viewState если нужны новые данные

### Шаг 3: Внесение изменений

**Правила безопасной модификации:**

1. **Не ломай существующие контракты:**
   - Не удаляй поля из ViewState без необходимости (добавляй новые)
   - Не удаляй события из ViewEvent (добавляй новые)
   - Не меняй сигнатуру UseCase.execute() если он используется в других местах

2. **Соблюдай архитектуру:**
   - Новая логика — только в ViewModel
   - Новый UI — только в View
   - Новые данные — через UseCase → Repository → DataSource
   - Каждый новый класс — в отдельном файле

3. **Обновляй DI:**
   - Новый UseCase → добавь bind/factory в DI модуль
   - Новый Repository → добавь bind/single в DI модуль
   - Новый DataSource → добавь bind/single в DI модуль

### Шаг 4: Валидация
- [ ] Существующий функционал не сломан
- [ ] Новый код следует архитектурным правилам
- [ ] Screen остаётся тонким адаптером
- [ ] View не содержит логики
- [ ] ViewModel не импортирует Compose
- [ ] Новые UseCase наследуют UseCase<Params, Result>
- [ ] DI модуль обновлён
- [ ] Каждый новый класс в отдельном файле

### Шаг 5: Вывод
1. **Summary** — что изменено и зачем
2. **Changed files** — список изменённых файлов с описанием изменений
3. **New files** — список новых файлов (если есть)
4. **Full code** — код всех изменённых и новых файлов
5. **Architecture validation** — подтверждение соблюдения правил
