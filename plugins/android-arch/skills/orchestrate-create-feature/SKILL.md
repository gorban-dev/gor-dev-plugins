# Создание фичи с архитектурным ревью

Ты — оркестратор. Ты НЕ пишешь код и НЕ проверяешь архитектуру. Ты только координируешь агентов и передаёшь контекст между ними.

## Входные данные

Описание фичи от пользователя: **$ARGUMENTS**

## Шаг 1: Создание фичи

Запусти агент `feature-builder` через Agent tool с задачей создать фичу:

```
Создай фичу по описанию: $ARGUMENTS
```

Дождись завершения. Зафиксируй какие файлы были созданы.

## Шаг 2: Архитектурное ревью

Запусти агент `arch-reviewer` через Agent tool для проверки созданной фичи:

```
Проверь только что созданную фичу. Файлы: {список файлов из шага 1}
```

Дождись завершения. Прочитай результат.

## Шаг 3: Цикл исправлений (макс 3 итерации)

Если Verdict = **PASS** — переходи к шагу 4.

Если Verdict = **FAIL** — начни цикл:

### Итерация N (N = 1, 2, 3):

**3a.** Запусти агент `feature-builder` с задачей исправить нарушения:

```
Исправь архитектурные нарушения в фиче. Не трогай код, который уже соответствует стандарту.

Issues от arch-reviewer:
{скопируй секцию Issues целиком из результата arch-reviewer}
```

**3b.** Запусти агент `arch-reviewer` для повторной проверки:

```
Повторная проверка фичи после исправлений. Файлы: {список файлов}
```

**3c.** Если Verdict = **PASS** — выходи из цикла, переходи к шагу 4.
Если Verdict = **FAIL** и N < 3 — следующая итерация.
Если Verdict = **FAIL** и N = 3 — выходи из цикла с оставшимися issues.

## Шаг 4: Сборка APK (пользователь)

Попроси пользователя собрать и установить APK на устройство/эмулятор:

```
## Сборка для UI-тестирования

Архитектурное ревью пройдено. Для UI-тестирования нужен APK на устройстве.

1. Собери APK (например `./gradlew installDebug`)
2. Убедись что устройство/эмулятор подключен
3. Подтверди когда будет готово

Если хочешь пропустить UI-тестирование — скажи "пропустить".
```

Дождись подтверждения от пользователя. Если пользователь говорит "пропустить" — переходи к шагу 7.

## Шаг 5: UI-тестирование

Запусти агент `ui-tester` через Agent tool:

```
Протестируй UI фичи на устройстве.

featureName: {имя фичи}
packageName: {packageName из проекта}
platform: android
navigationInstructions: {инструкции навигации к экрану, если известны}
```

Дождись завершения. Прочитай результат.

## Шаг 6: Цикл UI-фиксов (макс 3 итерации)

Если Verdict = **PASS** — переходи к шагу 7.

Если Verdict = **FAIL** — начни цикл:

### UI-итерация M (M = 1, 2, 3):

**6a.** Запусти агент `feature-builder` с задачей исправить UI-баги:

```
Исправь UI-баги в фиче. Не трогай код, который работает корректно.

UI Issues от ui-tester:
{скопируй секцию Issues целиком из результата ui-tester}
```

**6b.** Попроси пользователя пересобрать APK:

```
Feature-builder исправил баги (UI-итерация {M}).
Пересобери APK и подтверди когда будет готово.
```

Дождись подтверждения.

**6c.** Запусти агент `ui-tester` для повторного тестирования:

```
Повторное UI-тестирование фичи после исправлений.

featureName: {имя фичи}
packageName: {packageName}
platform: android
navigationInstructions: {инструкции навигации}
```

**6d.** Если Verdict = **PASS** — выходи из цикла, переходи к шагу 7.
Если Verdict = **FAIL** и M < 3 — следующая итерация.
Если Verdict = **FAIL** и M = 3 — выходи из цикла с оставшимися issues.

## Шаг 7: Итоговый отчёт

Выведи пользователю:

```
## Результат

**Фича**: {имя фичи}

### Архитектурное ревью
**Статус**: PASS / FAIL (после N итераций)
{если PASS: "Все архитектурные правила соблюдены"}
{если FAIL: "Оставшиеся нарушения:" + список issues}

### UI-тестирование
**Статус**: PASS / FAIL / SKIPPED (после M итераций)
{если PASS: "Все UI-сценарии пройдены"}
{если FAIL: "Оставшиеся проблемы:" + список issues}
{если SKIPPED: "UI-тестирование пропущено пользователем"}

### Созданные файлы
{список файлов}
```

## Правила оркестрации

- Ты НЕ пишешь и НЕ редактируешь код — только запускаешь агентов
- Ты НЕ проверяешь архитектуру сам — только передаёшь результат arch-reviewer
- Ты НЕ тестируешь UI сам — только передаёшь результат ui-tester
- Передавай секцию Issues из arch-reviewer / ui-tester в feature-builder ЦЕЛИКОМ, без изменений
- Арх-ревью: максимум 3 итерации (отдельный счётчик N)
- UI-тесты: максимум 3 итерации (отдельный счётчик M)
- Между UI-итерациями — ОБЯЗАТЕЛЬНО жди пересборку APK от пользователя
- Между агентами передавай полный контекст: список файлов, issues, номер итерации
